Sarthi — System Guide

How the whole thing fits together: the data model, the bottom-up learning flow, and — above all — exactly how the AI teaches, grades, translates and routes between models.

This is a living document, and it is admin-only. It lives at docs/architecture.html in the repo and is never published as a static URL — it is served solely through the Basic-auth'd /api/admin/guide endpoint and rendered in this Admin tab, so our design can't simply be copied. When we change how something works — a prompt, a grader, the model policy, the schema — update this file in the same change. It describes what is built in the code (read from source), not what is merely planned; sections that are partly built are tagged partial.

1 · What Sarthi is

Sarthi is an exam-prep maths tutor built on one idea: learn bottom-up. A subject is broken into chapters; each chapter into small concepts ("nodes") laid out bedrock-first. A learner can only reach a node once its prerequisites are passed. Each node is taught one-on-one by a Socratic AI tutor, then gated by checks the learner must clear to move on. When every chapter is done, the learner sits one clean past-paper exam.

Today's loaded corpora are maths: cbse10, cbse12, jee (Main + Advanced), plus a "Learn from Scratch" track (Kindergarten → Grade 8 → High School). The brand name in the UI is Sarthi.

Sister project. The same architecture powers "IE" (bottom_up_IE) for a different subject. This guide describes this repo (the maths/Sarthi build). They share the engine, not the content.

3 · Data model

Two halves: static content (the same for everyone) and learner state (per student). All in src/db/schema.ts.

Static content

  • chapters — id, title, subject, exam, and conceptOrder (the bottom-up ordering of its concepts).
  • concepts — the nodes. Each has role (bedrock/intermediate), order, prereqs[], a plain-words brief, the explanation (the tutor's only source of truth), keyMoves[] (the private checklist of ideas to elicit), misconceptions[], an optional advancedContent overlay (shown only on the advanced track), and needsAuthoring (Scratch-track placeholder nodes shown as "coming soon").
  • gates — the checks. Up to 5 authored slots per concept (sketch1, sketch2, explain, mcq, equation). Each gate has an answerType (mcq · symbolic · written · sketch), a grader (mcq · cas · rubric · vision), the expected spec (server-only, never serialised to the client), and a tier (foundation for everyone, or advanced).
  • bu_figure — textbook page images, each captioned by a vision pass and tagged with the concept(s) it illustrates, so the tutor can show the right figure inline.

Static content is loaded into memory once (2 queries) by src/core/content-cache.ts and served from there, so the hot paths — chapter list (0 DB reads) and node map (1 read, just the learner's statuses) — don't hammer the DB at scale. Call invalidateContentCache() after a re-import.

Learner state — event-sourced

Source of truth = the append-only event log (bu_event). The live dialogue, checklist progress and gate history are all reconstructed/derived from events. The other learner tables are fast derived views, not the truth.
TableWhat it holds
bu_eventAppend-only log: enter_node, tutor_turn, learner_turn, keymove_demonstrated, misconception_seen, gate_posed/answer/pass/fail, node_complete, chapter_complete, corpus_gap.
bu_node_performancePer-node status (locked → available → teaching → awaiting_gate → needs_reteach → passed), attempts, passes, fails, timing.
bu_node_checklistWhich key moves the learner has demonstrated, with the evidence line that proved each.
bu_gate_attemptEvery graded gate answer: prompt, answer, correct?, who graded it.
bu_chat_summaryPer-node running conversation summary + watermark (see §5.1 — bounds context growth).
bu_chapter_progressPer-chapter rollup (locked/active/complete, concepts passed/total).

bu_llm_callEvery model call: full request messages + full response + model/tokens/latency (see §5.3).

4 · Sequencing & gating

Logic in src/core/sequencer.ts.

  • Within a chapter: a node is available when all its in-chapter prerequisites are passed, else locked. Passing a node runs recomputeAvailabilityAfterPass() to unlock dependents (never downgrades a node already past available).
  • Across chapters: strict-linear. A chapter is complete when every concept in it is passed; the first not-complete chapter is active, everything after it is locked.
  • Every concept is teachable — each has at least one gate — so a chapter clears by passing each node's gate(s); full 5-gate authoring is not required to progress.

A node passes only when its entire gate set is cleared. On the advanced track, advanced-tier gates are appended after the foundation set (an advanced learner clears both); advanced gates are never shown to a foundation learner. A wrong answer gives targeted feedback and re-poses the same gate — no hard reset — and flips the node to needs_reteach for a fresh re-explanation.

5 · How the AI works

This is the heart of the system. The AI never invents curriculum — it teaches the one concept in front of it, grounded strictly in that concept's stored explanation. Everything below lives in src/core/ (node-agent.ts, teach-loop.ts, llm.ts, exam-profile.ts).

learner message→ build context (summary + recent turns)→ teachTurn — 1 LLM call→ apply checklist delta + log events→ server decides "ready for gate?"→ reply (translated if needed)
LLM calldeterministic server logicdatabase write

5.1 · The teaching loop built

Driven by respond() in teach-loop.ts, which calls teachTurn() in node-agent.ts.

  • One call, two jobs. A single LLM call returns JSON that holds both the next short Socratic tutor message and the checklist delta (which key moves the learner just demonstrated, with evidence). There is no separate "judge" call.
  • Instant opening. The first turn on entering a node is a deterministic, warm Socratic open (no LLM wait) — it names the topic and asks how the learner feels about it. The real teaching starts on the next turn.
  • The server owns completion, not the model. After applying the delta, the server checks whether all key moves are now demonstrated. That deterministic check — not the model's own readyForGate guess — decides when to offer the checks. If the model ends on a question right as the lesson completes, a deterministic warm closing is swapped in.
  • Bounded context. Rather than replaying the whole transcript, context = a stored running summary (bu_chat_summary) + only the turns since its watermark. On node entry, prior turns are folded into the summary (summarizeOnEntry, fire-and-forget). This keeps token cost flat no matter how long the conversation gets.
What the teaching prompt enforces (see the system prompt in buildMessages()):
  • Don't assume — refresh, then confirm-gate. Before using any term/symbol the learner hasn't shown in this conversation, refresh it in one plain sentence + a tiny example, ask one light confirmation question, then wait. The course is treated as self-contained — never "you learned this last year".
  • The Socratic loop as the default rhythm: surface ignorance → let them confess "I don't know" → fill the gap → return to the anchor.
  • Track tangents, return to the anchor — open one detour at a time, close it, name the bridge back.
  • One small step per message; never restate a fact as a question; ground every fact in the stored explanation (anti-drift).
  • If asked something the content genuinely lacks, don't invent — flag a corpusGap (logged to bu_corpus_gap for later review).
  • Exam-aware persona injected from exam-profile.ts (e.g. "a 15-year-old CBSE Class 10 student" vs "a JEE Advanced aspirant" vs a Kindergarten band) — so the tutor speaks and pitches at the right level.

5.2 · Grading the gates built

In answerGate() (teach-loop.ts), the grader is chosen by the gate's grader field. Deterministic where possible, LLM only where judgement is needed:

GraderHowLLM?
mcqNormalised string match against the correct option.No
casClean integer target → deterministic arithmetic (casEquivalent, handles prime-power products like 2^3·3^2). Otherwise LLM equivalence check (gradeEquation).Sometimes
rubricWritten answer judged against the rubric/ideal (gradeWritten) — passes on the core idea even with imperfect wording.Yes
visionHand-drawn answer read by a vision model against the rubric (gradeSketch).Yes

Grading reasoning runs in English (best quality); only the feedback is translated to the learner's language.

5.3 · The model router & cost policy built

HARD RULE (in llm.ts): real users (prod) are served by Claude Haiku; all testing uses NVIDIA NIM (free); offline falls back to a mock. Never run a test suite on Haiku — under NODE_ENV=test the router refuses Haiku and forces NIM/mock.
PurposeProd modelTest / free model
Tutoring + rubric/equation gradingClaude Haiku (claude-haiku-4-5)NIM meta/llama-3.3-70b-instruct
TranslationSarvam (Indic) → Google → Haiku/NIMsee §5.4
Vision (sketch help/grading)NIM nemotron-nano-12b-v2-vlsame
Figure captioningClaude Haiku vision—
Content authoringHaiku (bulk) / Opus (curated first nodes)—

The provider resolves from env (LLM_PROVIDER); default prefers free NIM when a key exists, else mock. NIM is called without response_format: json_object (it strands content as null for several models) — instead the prompt says "return ONLY JSON" and parseLooseJson() tolerantly extracts it.

Why every call is logged. recordLlmCall() writes the full request messages + full response + tokens + latency to bu_llm_call. That's the replay corpus for the NIM study (tools/nim-study.mjs): we can replay the exact prompts real users hit through candidate NVIDIA NIM models and compare quality against the captured Haiku baseline — the path toward a self-hosted-GPU switch — without ever re-calling a paid API. study is tooling/in-progress
API keys are for the human-facing product only. Do not call authoring/generation with a paid model flag from a script or agent. Content is authored interactively; teaching traffic is what spends on Haiku in prod.

5.4 · Language & voice built

translateText() in node-agent.ts. Content is stored once in English and translated dynamically per turn, so nothing goes stale. Maths ($…$), **bold** and emoji are masked so the translator can't mangle them. Provider ladder: Sarvam (best Indic quality, primary when keyed) → free Google endpoint → NIM fallback → original text. Voice (src/core/voice.ts, src/web/lib/voice.ts) adds read-aloud (TTS, on by default) and speak-to-type (STT); a one-time per-node invite offers Hindi / Punjabi / Tamil / Bengali.

5.5 · Vision & figures built

Two vision uses: (1) the scratchpad "Help me" reads the learner's handwritten working and gives one grounded hint (helpWithSketch → nimVision); (2) sketch gates are graded the same way. Separately, a Haiku-vision pass captions extracted textbook page images into bu_figure; the tutor may surface one inline when it genuinely helps (figureRef in the turn JSON), with a deterministic safety net if the learner explicitly says "show me the graph".

6 · The final exam built

After the chapters, the learner sits a real past paper (logic in src/core/papers.ts, answers in bu_paper_attempt). Subjective questions are marked by gradeSubjective() as an official board examiner would: partial / step marks out of the question's maximum, following the marking scheme — a half-right answer earns half the marks. If the grader is unavailable it awards 0 and says so rather than inventing marks. Every question carries its source node, so wrong answers roll straight up into a weak-concept review.

7 · Frontend

SPA in src/web/. App.tsx holds the view state (home · nodes · node · paper) plus two hash routes: #admin (operator dashboard) and #docs (public documentation hub). A learner gets an anonymous learnerId (localStorage UUID) so the home map renders before login; once logged in, the account id is the learnerId, so all progress ties to the account. Clicking into a chapter while logged out opens the auth modal, then replays the intended navigation.

Key components: Home, ChapterMap (the node graph), NodeView (the chat), NodeDetails (the live "Tutor's notes" panel), Scratchpad (handwriting canvas), PaperView (final exam), Documentation and AdminDashboard. Maths renders via MathText (KaTeX).

8 · Admin & observability

API in src/api/admin.ts, UI in src/web/components/AdminDashboard.tsx, reached at #admin. Read-only operator views gated by HTTP Basic auth (ADMIN_USER / ADMIN_PASSWORD); fail-closed — if those env vars are unset, every admin route returns 503. Credentials live only in sessionStorage, never in the bundle.

Panels: headline KPIs (users, nodes passed, gate pass rate, LLM calls), LLM usage by provider/model/purpose (the Haiku-vs-NIM picture), traffic + signup funnel, recent conversation transcripts, and the raw LLM-call log that feeds the NIM study. This System Guide is the second tab here.
